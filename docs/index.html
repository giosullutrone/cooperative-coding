What is CooperativeCoding?

Current agentic coding treats software as text to be generated. CooperativeCoding treats it as architecture to be negotiated.

CooperativeCoding is an open standard for human-agent collaborative software design. It provides a shared data model built on JSON Canvas v1.0 where architectural elements — classes, methods, fields, packages, and their relationships — are first-class objects of collaborative design stored in a ccoding metadata namespace.

Code design and code implementation are different skills that benefit from different tools. Humans excel at architectural thinking — defining responsibilities, drawing boundaries, seeing the big picture. Agents excel at translating precise specifications into correct, complete code. Today, both activities happen in the same text editor, forcing the human to think at the wrong level of abstraction and the agent to guess at the right one.

CooperativeCoding separates these concerns. Both human and agent work on a visual canvas — a simplified UML where design elements carry documentation contracts. The agent also works on the code, implementing the canvas design and keeping canvas and code in bidirectional sync. They cooperate through ghost proposals: agent-created elements appear as tentative suggestions that the human reviews, accepts, or rejects before anything gets implemented.

The specification is language-agnostic and tool-agnostic. It extends JSON Canvas with typed nodes, semantic edges, a status lifecycle, and sync rules. Concrete implementations require a canvas tool plugin, a language binding, and an agent integration — but the core spec defines the universal contract that all of them share.

Seven Principles

The foundation of human-AI collaborative design.

01

Design Authority is Human

Software architecture demands judgment that emerges from business context, team culture, and lived experience. The human always has final say. The agent proposes, the human disposes. No design change is applied without explicit human approval.

02

Progressive Detail

Not everything deserves the same level of attention. Class nodes give the overview; only architecturally significant methods and fields are promoted to their own detail nodes. The canvas highlights what matters, not every line of code.

03

Documentation as Contract

The documentation block is the bridge between canvas and code. Responsibility statements, pseudo code, and constraints form the contract that the agent implements against and the sync engine uses to detect drift.

04

Bidirectional Truth

Canvas and code are two views of the same system. Neither is primary. Changes in either propagate through the sync engine. When they disagree, conflict detection ensures a human decides — no silent overwrites.

05

Visual Simplicity

The canvas shows a simplified UML focused on architectural thinking. A handful of node kinds, a handful of edge relations, and structured markdown for content. A place to see the shape of the system at a glance.

06

Any Entry Point

Sketch on a blank canvas. Describe a system in natural language and let the agent propose the design. Import existing code and let the sync engine generate the canvas. The paradigm meets you where you are.

07

Language-Agnostic, Tool-Agnostic

Nodes, edges, ghost proposals, and the cooperation lifecycle are universal. Each programming language gets a binding, each canvas tool gets an implementation, each agent gets an integration. The core spec defines the contract; bindings fulfill it.

The Cooperation Loop

A continuous cycle where human intent and agent capability meet.

Design

Human architect creates or updates the canvas — classes, methods, fields, relationships, and documentation contracts. The agent may also propose design changes as ghost nodes. This is the authoritative moment for architectural decisions.

Review

The human reviews ghost proposals, accepting elements that fit the vision and rejecting those that do not. Ghosts may be edited before acceptance. Only accepted elements proceed to implementation.

Implement

The agent generates or updates source code based on accepted canvas elements. Documentation blocks become docstrings, edges become inheritance declarations and imports, node kinds become class skeletons.

Sync

The sync engine detects changes on both sides and propagates them. Code changes update the canvas; canvas changes update the code. Conflicts are detected and surfaced for human resolution. The loop begins anew.

Data Model

The building blocks of architectural description — a semantic layer on top of JSON Canvas v1.0.

Node Kinds

class A class, struct, trait, or equivalent named type. The primary building block. Carries responsibility, fields, methods, and constraints in its documentation block.
method A method or function promoted to its own detail node. Connected to its parent class via a detail edge. Contains signature, pseudo code, and constraints.
field A field or property promoted to its own detail node. Used when a field carries significant design semantics — a configuration object, a dependency injection point.
package A package, module, or namespace that groups related types. Used for high-level architectural views and directory structure generation.
test A test class or suite that verifies production code. Connected to classes under test via tests edges. Carries test execution results alongside design intent.
interface An interface, protocol, or abstract contract. May also be represented as a class node with an appropriate stereotype.
module A single-file module or compilation unit, distinct from package in languages where the distinction matters.

Edge Relations

inherits Class extends or subclasses another. Generates inheritance declarations.
implements Class adopts an interface or protocol. Generates conformance declarations.
composes Has-a relationship. The edge label carries the field name. Generates typed field declarations.
depends Import-level dependency. Generates import statements in the source file.
calls Runtime method invocation. Informational — not synced. Documents runtime flow.
detail Links a class node to a promoted method or field. Structural, not independently synced.
tests Test node verifies a class. Propagates staleness when the class changes.
context Links a context node to a code element. Canvas-only — design rationale and references.
Stereotypes extend node semantics with language-specific subtypes. The set is open and extensible: protocol, dataclass, enum, abstract, singleton, mixin, and any others your language binding defines. Unknown stereotypes are preserved and displayed as-is — implementations must not reject them.

Status Lifecycle

Four states. Five transitions. A clear state machine governing every element on the canvas.

proposed
A ghost — a tentative design element awaiting human review. Exists on the canvas as a visual suggestion but has no authority over the codebase. Not synced to code. Canvas tools render ghosts with reduced opacity or dashed borders.
accepted
Part of the canonical design. Represents a deliberate architectural decision confirmed by a human. Synced to code during both canvas-to-code and code-to-canvas sync. Rendered with full visual weight.
rejected
A proposal that a human explicitly declined. Remains in the canvas file as a record of the decision but is not part of the active design. Not synced to code. Hidden by default or visually suppressed.
stale
The element's corresponding code was deleted, moved, or renamed in a way the sync engine could not resolve. The canvas node remains, but its link to the codebase is broken. Awaiting human resolution. Not auto-deleted.

Bidirectional Sync

Canvas and code are two views of the same system. The sync engine keeps them honest.

What Syncs

  • New accepted class nodes generate source files with class skeletons
  • Changed field or method lists update class definitions in code
  • Documentation edits propagate to docstrings and doc comments
  • New inherits edges create inheritance declarations
  • New implements edges create protocol conformance
  • New composes edges generate typed field declarations
  • New depends edges generate import statements
  • Code-side class creation generates canvas nodes
  • Code-side documentation changes update canvas node text
  • Test results flow from code to test node content

What Never Syncs

  • Ghost (proposed) elements — canvas-only proposals
  • Rejected elements — preserved for decision history only
  • Stale elements — awaiting human resolution
  • Method bodies — implementation is code's domain
  • Local variables and control flow within functions
  • Context nodes — design rationale stays on the canvas
  • calls edges — informational, not structural
  • Canvas layout positions — node arrangement is visual only
Conflict Resolution: When both canvas and code change the same element between sync cycles, the sync engine detects the conflict and refuses to silently overwrite either side. Implementations present both versions to the human for resolution — three-way merge, side selection, or deferral. Conflicts are detected at the element level, not the file level: changing ClassA on the canvas and ClassB in the same source file is not a conflict.

Getting Started

Four steps to begin working with CooperativeCoding.

1

Read the Specification

Start with the Introduction for the vision and principles. Then explore the Data Model, Lifecycle, Sync, and Language Bindings documents for the full technical contract.

Read the Introduction →
2

Choose Your Canvas Tool

CooperativeCoding works with any canvas tool that supports JSON Canvas v1.0. The reference implementation uses Obsidian with a dedicated plugin that renders ghost nodes, detail edges, and status indicators.

Explore JSON Canvas →
3

Set Up a Language Binding

Each programming language needs a binding that maps abstract spec concepts to language-specific idioms. The Python binding is the reference implementation, covering stereotypes, AST parsing, type translation, and sync mapping.

Python Binding Reference →
4

Integrate Your Agent

Teach your agentic coding tool the cooperation protocol: propose design changes as ghost nodes, implement accepted elements, and keep canvas and code in sync. The spec defines the cooperation contract; your agent fulfills it.

About

CooperativeCoding is an open specification for human-AI collaborative software design. It separates architectural thinking from code implementation, giving each the right tool and the right authority.
